# Output Schema

Use this schema when the user needs structured output, durable topic-bank records, or Coze workflow integration.

## Benchmark Account

- `name`: account or benchmark label.
- `platform`: default to `小红书` when unspecified.
- `signal`: why this benchmark matters, such as recent growth, repeated high-engagement structure, or clear audience fit.
- `content_patterns`: reusable patterns in titles, hooks, body structure, proof, CTA, or posting rhythm.
- `visual_patterns`: cover, color, typography, image composition, page structure, or screenshot style.
- `risk_notes`: what not to copy or what needs verification.

## Demand Insight

- `cluster`: demand category, such as beginner barrier, tool anxiety, time constraint, price sensitivity, or resource request.
- `user_words`: short anonymized user phrases or comments.
- `pain`: concrete pain behind the comments.
- `desired_outcome`: what users want to achieve.
- `content_angle`: how to turn the demand into content.
- `confidence`: `high`, `medium`, or `low`, based on evidence quality.

## Topic Record

- `title`: working note title.
- `audience`: target reader.
- `hook`: first-page or first-sentence hook.
- `demand_source`: comment cluster or benchmark pattern that justifies the topic.
- `outline`: 3-6 bullets for the note.
- `proof_needed`: data, screenshot, case, template, checklist, or personal experience needed before publishing.
- `differentiation`: what makes this not a copy of the benchmark.
- `priority`: `high`, `medium`, or `low`.
- `expected_view_score`: 0-100 score for likely Xiaohongshu view potential.
- `view_evidence`: views, saves, comments, search terms, hot-word notes, or user-selection evidence.
- `selected`: whether this topic enters the current production chain.
- `selection_reason`: why the workflow selected it.

## Editable Prompt

- `key`: stable prompt key, such as `openai_text_description`, `grok_expert_image_set`, or `openai_text_skill.generate_xhs_text`.
- `title`: human-readable prompt title for online editing.
- `target_model`: model or node that consumes the prompt.
- `default_prompt`: default prompt generated by the workflow.
- `final_prompt`: user-overridden prompt if `prompt_overrides` is provided, otherwise the default.
- `editable`: whether the prompt should be exposed in UI.

## Skill Subflow

Use `skill_subflows` to show OpenAI and Grok as visual, editable subflows instead of opaque model calls.

- `skill_key`: stable subflow key, such as `openai_text_skill` or `grok_image_skill`.
- `title`: visible subflow title.
- `provider`: `openai` or `grok`.
- `model`: configured model for this subflow.
- `mode`: reasoning or generation mode.
- `endpoint`: real API endpoint used when `execute_model_calls=true`.
- `description`: what the subflow does.
- `editable`: whether operations can edit the subflow.
- `steps`: ordered `SkillSubflowStep` records.
- `status`: `ready`, `dry_run`, `completed`, or `failed`.

Each `SkillSubflowStep` includes:

- `step_key`: stable step key.
- `title`: visible step title.
- `node_type`: `rule`, `prompt`, `model`, or `review`.
- `model_or_tool`: model or local skill used by the step.
- `prompt_key`: editable prompt key, such as `grok_image_skill.compose_page_prompts`.
- `default_prompt`: default step prompt.
- `final_prompt`: prompt after `skill_flow_overrides` or `prompt_overrides`.
- `input_keys`: upstream fields consumed by the step.
- `output_keys`: fields produced by the step.
- `editable`: whether the step can be changed.
- `enabled`: whether this step is included in the model input.
- `notes`: editing guidance for operators.

`skill_flow_overrides` may override a full subflow or individual steps:

```json
{
  "openai_text_skill": {
    "model": "gpt-5.5",
    "mode": "ultra_high",
    "steps": {
      "generate_xhs_text": {
        "prompt": "Use a more practical operations voice.",
        "enabled": true
      }
    }
  },
  "grok_image_skill": {
    "steps": {
      "compose_page_prompts": {
        "prompt": "Make a 3:4 cartoon image set with one action per page."
      }
    }
  }
}
```

For operators who do not edit JSON, prefer these plain GraphInput fields:

- `openai_text_skill_prompt`: maps to `openai_text_skill.steps.generate_xhs_text.prompt`.
- `grok_image_skill_prompt`: maps to `grok_image_skill.steps.compose_page_prompts.prompt`.
- `grok_visual_rules_prompt`: maps to `grok_image_skill.steps.load_visual_rules.prompt`.
- `image_aspect_ratio`: updates the image rule and Grok request aspect ratio.
- `image_style`: updates the image rule and Grok prompt style.
- `openai_review_enabled`: toggles `openai_text_skill.steps.review_text_for_image.enabled`.
- `grok_review_enabled`: toggles `grok_image_skill.steps.review_set_consistency.enabled`.

## Workflow Diagram

Use `workflow_diagram_nodes` and `workflow_diagram_edges` to render the workflow as a visual graph.

Each node includes:

- `node_key`: stable graph node key.
- `title`: visible node title.
- `node_type`: `workflow`, `subflow`, `step`, `start`, or `end`.
- `lane`: visual group or swimlane.
- `subflow_key`: parent subflow for child steps.
- `row` and `column`: suggested layout position.
- `editable`: whether the UI should expose an edit affordance.
- `input_keys` and `output_keys`: fields shown on the node.

Each edge includes:

- `from_node`: source node key.
- `to_node`: target node key.
- `label`: optional edge label.
- `edge_type`: `sequence`, `parallel`, `join`, or `subflow`.

The main graph should include these parallel and join edges:

- `prompt_editor -> openai_text` with `edge_type=parallel`.
- `prompt_editor -> grok_image_set` with `edge_type=parallel`.
- `openai_text -> finalize` with `edge_type=join`.
- `grok_image_set -> finalize` with `edge_type=join`.

## Operator Edit Panel

Use `operator_edit_panels` for non-technical operators. Each panel has:

- `panel_key`: target skill, such as `openai_text_skill` or `grok_image_skill`.
- `title`: visible panel title.
- `description`: short operator-facing guidance.
- `controls`: form controls.

Each control has:

- `control_key`: stable control key.
- `label`: visible label.
- `control_type`: `text`, `textarea`, `select`, `toggle`, or `number`.
- `input_key`: plain GraphInput field to bind.
- `current_value`: current form value.
- `placeholder`: writing hint.
- `options`: choices for selects.
- `target_path`: where the field maps inside a subflow.
- `help_text`: plain-language operator help.

## Image Style Rule

- `aspect_ratio`: default `3:4`.
- `style`: default `cartoon`.
- `reference_image_notes`: user-provided visual observations, such as character, palette, line, lens, mood, and composition.
- `reference_image_urls`: optional reference URLs.
- `must_have`: required visual rules.
- `avoid`: negative rules.
- `consistency_rules`: rules that make all images feel like one set.

## OpenAI Text Package

- `provider`: `openai`.
- `model`: default `gpt-5.5`.
- `reasoning_mode`: default `ultra_high`; API payload maps this to `reasoning.effort=xhigh`.
- `request`: dry-run request plan or real-call summary. It should include `skill_key=openai_text_skill`; API keys must never appear in `payload`.
- `title_options`: cover title options.
- `post_description`: Xiaohongshu caption/body.
- `card_script`: page-by-page text script.
- `image_brief`: visual brief passed to Grok.
- `status`: `dry_run`, `ready`, `completed`, or `failed`.

## Grok Image Set

- `provider`: `grok`.
- `model`: default `grok-imagine-image-quality`.
- `mode`: default `Expert`.
- `aspect_ratio`: default `3:4`.
- `style`: default `cartoon`.
- `images`: ordered image generation records.
- `consistency_rules`: set-level visual consistency rules.
- `status`: `dry_run`, `completed`, `partial_failed`, or `failed`.

Each image generation record includes:

- `page`: integer.
- `headline`: short page title.
- `prompt`: Grok Expert prompt.
- `aspect_ratio`: default `3:4`.
- `style`: default `cartoon`.
- `request`: dry-run or executable request plan.
- `request.skill_key`: should be `grok_image_skill`.
- `image_url`: optional real generated image URL.
- `status`: `dry_run`, `completed`, or `failed`.

## Card Package

- `topic_title`: selected topic.
- `cover_options`: 3-5 cover headline options.
- `cards`: ordered page specs. Each page should include:
  - `page`: integer.
  - `headline`: short page title.
  - `body`: concise copy.
  - `visual_prompt`: visual direction or image-generation prompt.
- `caption`: full Xiaohongshu body copy.
- `hashtags`: 5-10 tags.
- `cta`: a soft next action.
- `review_checklist`: factual, ethical, and style checks before publishing.

## Workflow Step

- `node_key`: graph node key.
- `title`: visible node title.
- `model_or_tool`: model, tool, or skill used by the node.
- `prompt_key`: prompt key if the node consumes editable prompts.
- `output_keys`: important output fields.
- `status`: node status.

## Next Commands

Write 3-5 short commands the user can paste next, such as:

- `继续扩写第 1 个选题，生成 6 页卡片。`
- `用更适合新手的语气改写封面标题。`
- `把评论需求整理成 Obsidian 选题库 Markdown。`
